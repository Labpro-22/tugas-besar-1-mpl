1. Arsitektur Sistem (Layered Architecture)
Program Nimonspoli dibangun dengan tiga lapis tanggung jawab yang terpisah, sesuai prinsip Layered
Architecture dan Single Responsibility dari SOLID:


      Layer                 Kelas Utama                             Tanggung Jawab

 User               GameUI,      BoardRenderer,       Seluruh input/output terminal. Tidak ada
 Interaction (UI)   CommandParser,                    logika permainan di layer ini.
                    PropertyCardRenderer

 Game      Logic    GameEngine, Board, Player,        Seluruh aturan permainan, kalkulasi sewa,
 (Core)             Tile     (dan    turunannya),     lelang, festival, kebangkrutan, manajemen
                    ActionCard (dan turunannya),      state.
                    SkillCard (dan turunannya),
                    TurnManager,
                    AuctionManager,
                    BankruptcyHandler,      Dice,
                    CardDeck<T>, GameContext

 Data Access        ConfigLoader, SaveManager,        Baca file konfigurasi, simpan/muat state
                    TransactionLogger,                permainan, catat log transaksi.
                    PropertyConfig,     TaxConfig,
                    SpecialConfig,     MiscConfig,
                    ConfigData,          LogEntry,
                    PlayerState,    PropertyState,
                    GameState




2. Penerapan Konsep OOP Wajib

     Konsep OOP                              Implementasi dalam Nimonspoli

 Inheritance         &   Tile → PropertyTile → StreetTile/RailroadTile/UtilityTile; Tile → ActionTile
 Polymorphism            →
                         GoTile/JailTile/GoToJailTile/FreeParkingTile/TaxTile/CardTile/FestivalTile;
                         ActionCard                                                               →
                         MoveToNearestStationCard/MoveBackCard/GoToJailCard/BirthdayCard/
                         DoctorFeeCard/CampaignCard;                    SkillCard                 →
                         MoveCard/DiscountCard/ShieldCard/TeleportCard/LassoCard/Demolition
                         Card. Metode onLanded(), calculateRent(), execute(), use() dipanggil
                         secara polimorfik.

 Abstract Class      &   Tile: onLanded() pure virtual. PropertyTile: calculateRent() pure virtual,
 Virtual Function        getSellValueToBank() virtual. ActionCard: execute() pure virtual.
                         SkillCard: use() pure virtual, getTypeName() pure virtual.
 Generic         Class   CardDeck<T> mengelola tumpukan kartu apapun. GameContext
 (Template)              menyimpan CardDeck<ActionCard> (dua instans, satu untuk
                         Kesempatan dan satu untuk Dana Umum) dan CardDeck<SkillCard>
                         untuk Kemampuan Spesial.

 Exception Handling      NimonException (abstract, extends std::exception) sebagai base.
                         Subclass konkret: InsufficientFundsException, CardHandFullException,
                         InvalidPropertyException, InvalidCommandException. Di-throw oleh
                         Game Logic Layer, di-catch oleh GameEngine atau BankruptcyHandler.

 Function                Board::getTile(int    index)     dan  Board::getTile(string    code).
 Overloading             Board::getProperties()       dan     Board::getProperties(ColorGroup
                         colorGroup).          GameUI::showLog(vector<LogEntry>)          dan
                         GameUI::showLog(vector<LogEntry>, int n).

 Operator                Player::operator+=, operator-= untuk transaksi saldo (operator-=
 Overloading             melempar InsufficientFundsException jika balance tidak cukup).
                         operator<, operator> untuk membandingkan total kekayaan pada
                         penentuan pemenang MAX_TURN.

 STL — Vector            tiles di Board; properties dan hand di Player; entries di
                         TransactionLogger; turnOrder di TurnManager; deck dan discard di
                         CardDeck<T>.

 STL — Map               propertyIndex       (string→PropertyTile*)        di    Board; colorMap
                         (TileCategory→string) di BoardRenderer; rentTable (int→int) di
                         RailroadTile; multiplierTable (int→int) di UtilityTile.




3. Hierarki Kelas
Hierarki kelas lengkap setelah revisi dan pembersihan redundansi:


  [TILE HIERARCHY]
  Tile (abstract)
    ├── PropertyTile (abstract)
      │   ├── StreetTile
      │   ├── RailroadTile
      │   └── UtilityTile
    └── ActionTile (abstract)
          ├── GoTile
          ├── JailTile
          ├── GoToJailTile
          ├── FreeParkingTile
          ├── TaxTile
          ├── CardTile
          └── FestivalTile

  [CARD HIERARCHY]
  ActionCard (abstract)
    ├── MoveToNearestStationCard
    ├── MoveBackCard
    ├── GoToJailCard
    ├── BirthdayCard
    ├── DoctorFeeCard
    └── CampaignCard
  SkillCard (abstract)
    ├── MoveCard
    ├── DiscountCard
    ├── ShieldCard
    ├── TeleportCard
    ├── LassoCard
    └── DemolitionCard

  [EXCEPTION HIERARCHY]
  std::exception (std library)
    └── NimonException (abstract base)
          ├── InsufficientFundsException
          ├── CardHandFullException
          ├── InvalidPropertyException
          └── InvalidCommandException

  [GENERIC]
  CardDeck<T>   (T = ActionCard* | SkillCard*)

  [GAME LOGIC — CONCRETE]
  Player, Board, Dice, TurnManager, AuctionManager
  BankruptcyHandler, GameContext, GameEngine

  [DATA ACCESS — CONCRETE]
  PropertyConfig, TaxConfig, SpecialConfig, MiscConfig, ConfigData
  LogEntry, PlayerState, PropertyState, GameState
  ConfigLoader, SaveManager, TransactionLogger

  [UI — CONCRETE]
  GameUI, BoardRenderer,     CommandParser,    PropertyCardRenderer,     Command

4.2 Exception Classes
Seluruh exception menggunakan enum TaxType dan CardType — keduanya aman karena hanya dipakai sebagai
label data (bukan dispatch logika switch besar). Exception di-throw oleh Game Logic Layer, di-catch oleh
GameEngine atau BankruptcyHandler.
NimonException [ABSTRACT BASE]
extends: std::exception

Tujuan: Kelas dasar abstrak bagi seluruh exception Nimonspoli. Menyediakan atribut message
dan override what().

5. Catatan Desain
→ Enum sebagai label data, bukan dispatcher logika
Enam enum digunakan dalam desain ini (TileCategory, ColorGroup, PropertyStatus, PlayerStatus, TaxType,
CardType). Semuanya berperan sebagai label/state machine, bukan sebagai pengontrol switch-case besar.
TaxType dan CardType masing-masing hanya memiliki dua nilai dan digunakan dalam if-else sederhana — tidak
ada risiko arsitektur prosedural.



→ Konsistensi tipe enum
ColorGroup digunakan secara konsisten: StreetTile::colorGroup bertipe ColorGroup, StreetTile::getColorGroup()
mengembalikan        ColorGroup,     Board::getProperties(ColorGroup)    menerima       ColorGroup,      dan
Player::isMonopolizing(ColorGroup, ...) menerima ColorGroup. TileCategory digunakan secara konsisten di
Tile::category dan BoardRenderer::colorMap.



→ Penghapusan operator+ dan operator-
Operator + dan - (non-compound) pada Player dihapus karena redundant. Seluruh transaksi saldo cukup dilayani
oleh operator+= dan operator-=. Tidak ada kebutuhan membuat objek Player baru dari operasi aritmetika.



→ Penghapusan sellAllBuildings()
Metode sellAllBuildings() dihapus dari StreetTile karena merupakan loop yang memanggil sellBuilding() berulang.
Logika ini menjadi tanggung jawab BankruptcyHandler yang memanggil sellBuilding() dalam loop hingga
buildingLevel == 0, menghindari duplikasi tanggung jawab.



→ Pemisahan moveTo dan gaji GO
Player::moveTo(newIndex) hanya memperbarui posisi dan mengembalikan bool passedGo. GameEngine yang
memanggil GoTile::awardSalary(player) jika passedGo == true. Ini menghindari Player bergantung pada GoTile
secara langsung (cross-dependency antara Player dan GoTile).



→ Pemisahan deck Kesempatan dan Dana Umum
GameContext menyimpan dua instans CardDeck<ActionCard> terpisah: chanceDeck dan communityDeck.
CardTile memilih deck berdasarkan cardType (if-else dua cabang). Kartu-kartu ActionCard tidak perlu mengetahui
kategorinya — pemisahan ditangani di tingkat deck dan CardTile.



→ GameContext sebagai dependency container
Tile::onLanded(), ActionCard::execute(), dan SkillCard::use() menerima GameContext& sebagai parameter
tunggal. Ini menghindari tanda tangan metode yang panjang dan mengurangi coupling langsung antara Tile/Card
dan GameEngine.



→ Tidak ada struct
Seluruh container data (PropertyConfig, TaxConfig, SpecialConfig, MiscConfig, ConfigData, LogEntry,
PlayerState, PropertyState, GameState, Command) diimplementasikan sebagai kelas dengan atribut privat dan
getter — tidak ada struct sama sekali.
